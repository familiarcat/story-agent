# 📁 .md File Organization & RAG Integration System

## PURPOSE

This document instructs all systems (AI assistants, automation, developers) on how to properly organize markdown files in the story-agent monorepo and integrate them with the RAG (Retrieval-Augmented Generation) system.

**Goals**:
- Maintain consistent .md file organization
- Enable efficient RAG retrieval
- Prevent .md files from accumulating in the root
- Create discoverable, well-indexed documentation
- Support multi-phase project execution workflow

## ⚙️ AUTOMATIC ORGANIZATION WORKFLOW

When generating new `.md` files:

### STEP 1: Determine Classification
Ask yourself: **"What is the PRIMARY purpose of this document?"**

Examples:
- "Explains how to organize files" → `docs/prompts/system-prompts/`
- "Lists todos for current session" → `docs/status/todos/`
- "Summarizes session completion" → `docs/status/sessions/`
- "Guides users on setup" → `docs/guides/setup/`
- "Explains technical architecture" → `docs/knowledge/architecture/`

### STEP 2: Select Target Folder
Choose from the hierarchy above based on classification.

### STEP 3: Create Subdirectory (if needed)
```bash
mkdir -p docs/{folder}/{subfolder}/
```

### STEP 4: Name File Appropriately
Use pattern: `{descriptor}-{version-or-date}.md` or `{descriptor}.md`

Examples:
- `docs/status/sessions/2026-06-07-session-summary.md`
- `docs/status/todos/active-todos.md`
- `docs/knowledge/architecture/microservices-overview.md`
- `docs/guides/quick-start/setup-guide.md`

### STEP 5: Add to Index
Update the appropriate `docs/` subfolder's `README.md` or index:
```markdown
## Documentation Index

### Sessions & Status
- [2026-06-07 Session](status/sessions/2026-06-07-session-summary.md)
- [Active Todos](status/todos/active-todos.md)

### Knowledge
- [Microservices Overview](knowledge/architecture/microservices-overview.md)
```

### STEP 6: Tag for RAG Indexing
Add frontmatter metadata:
```yaml
---
title: "Document Title"
category: "prompts"  # or knowledge, guides, crew, etc
tags: ["tag1", "tag2"]
searchable: true
version: "1.0"
---
```

## 🤖 RAG INTEGRATION REQUIREMENTS

### For Each Document

**Required Frontmatter**:
```yaml
---
title: "Clear, descriptive title"
category: "folder_name"  # prompts, knowledge, guides, crew, phases, etc
tags: ["relevant", "keywords", "for", "search"]
searchable: true
version: "1.0"
last_updated: "2026-06-07"
---
```

**RAG Indexing Rules**:
1. ✅ All docs under `docs/` are auto-indexed
2. ✅ Only documents with `searchable: true` are RAG-queryable
3. ✅ Use meaningful categories for retrieval grouping
4. ✅ Tag documents consistently for cross-references
5. ✅ Keep titles clear and descriptive
6. ✅ Update version when content changes significantly

### RAG Query Examples

```
# Search by category
"Find all prompts related to crew coordination"
→ Searches docs/prompts/ and docs/crew-prompts/

# Search by tag
"Show me all documents tagged 'architecture'"
→ Returns docs/knowledge/architecture/*.md

# Search by type + keyword
"Guides for setting up Supabase"
→ Searches docs/guides/ for "supabase"

# Search by phase
"Phase 1 execution instructions"
→ Searches docs/phases/phase-1-execution/
```

## 🚫 NEVER DO THIS

### ❌ DO NOT create .md files in these locations:
- ❌ Root directory (except README.md, START_HERE.md)
- ❌ `packages/` subdirectories (except specific .md files per package)
- ❌ `projects/` subdirectories (except project-level README.md)

### ❌ DO NOT:
- ❌ Add 20 files to root at once
- ❌ Create docs without frontmatter metadata
- ❌ Use vague filenames ("doc.md", "notes.md")
- ❌ Forget to update index files
- ❌ Leave documents untagged or unsearchable

## ✅ BEST PRACTICES

### 1. Name Files Descriptively
```
✅ GOOD:  crew-manifest-v2.md
❌ BAD:   crew.md

✅ GOOD:  phase-1-execution-checklist.md
❌ BAD:   todo.md

✅ GOOD:  mcp-tools-api-reference.md
❌ BAD:   api.md
```

### 2. Use Consistent Frontmatter
```yaml
---
title: "Descriptive Title Here"
category: "knowledge"  # Use folder name
tags: ["architecture", "database", "design"]
searchable: true
version: "1.0"
last_updated: "2026-06-07"
related_docs:
  - "docs/knowledge/architecture/other-file.md"
---
```

### 3. Group Related Files
```
✅ All crew docs in docs/crew/
✅ All phase docs in docs/phases/
✅ All setup guides in docs/guides/setup/
```

### 4. Keep Index Files Updated
```markdown
## Documentation Index

### Crew Documentation
- [Crew Manifest](crew/manifests/crew-manifest.md)
- [Crew Roles](crew/roles/captain-role.md)

### Knowledge Base
- [Architecture](knowledge/architecture/overview.md)
- [Database Schema](knowledge/database/schema.md)
```

### 5. Link Between Documents
```markdown
See also: [Related document](../path/to/document.md)
```

## 🔍 VERIFICATION

After organizing files:

1. **Link Check**: All links still work
   ```bash
   grep -r "^\[.*\](.*\.md)" docs/ | while read line; do echo "$line"; done
   ```

2. **Frontmatter Check**: All docs have metadata
   ```bash
   for f in docs/**/*.md; do grep -q "^---" "$f" || echo "Missing: $f"; done
   ```

3. **Index Check**: All files listed in appropriate index
   ```bash
   grep -r "$(basename $f)" docs/*/README.md
   ```

4. **RAG Validation**: Documents are searchable
   - Check `searchable: true` in frontmatter
   - Verify tags are meaningful
   - Confirm category matches folder

## 📞 QUESTIONS & ANSWERS

**Q: Where should session completion reports go?**  
A: `docs/status/completions/` with format: `{phase-or-project}-completion-{date}.md`

**Q: Can I store MCP prompts anywhere?**  
A: Yes, keep `packages/mcp-server/PROMPT_ENGINEERING.md` in package, but general prompts go in `docs/prompts/`

**Q: How do I cross-reference documents?**  
A: Use relative links: `[Title](../other-folder/file.md)` and add to `related_docs:` in frontmatter

**Q: What if my document fits multiple categories?**  
A: Store in primary category, add cross-references and tags for secondary categories

**Q: How are documents discovered for RAG?**  
A: Via `docs/` folder scan + frontmatter metadata (category, tags, searchable flag)
